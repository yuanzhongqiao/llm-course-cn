<p dir="auto" _msttexthash="1724514350" _msthash="388">NLP 是人工智能的一个迷人分支，它弥合了人类语言和机器理解之间的差距。从简单的文本处理到理解语言的细微差别，NLP 在许多应用程序中发挥着至关重要的作用，例如翻译、情感分析、聊天机器人等等。</p>
<ul dir="auto">
<li _msttexthash="1457178346" _msthash="389"><strong _istranslated="1">文本预处理</strong>：了解各种文本预处理步骤，如分词（将文本拆分为单词或句子）、词干提取（将单词减少为根形式）、词形还原（类似于词干提取，但要考虑上下文）、停用词删除等。</li>

<li _msttexthash="536461809" _msthash="391"><strong _istranslated="1">单词嵌入</strong>：单词嵌入是一种单词表示形式，它允许具有相似含义的单词具有相似的表示形式。主要方法包括 Word2Vec、GloVe 和 FastText。</li>
<li _msttexthash="870942592" _msthash="392"><strong _istranslated="1">递归神经网络 （RNN）：</strong>了解 RNN 的工作原理，RNN 是一种旨在处理序列数据的神经网络。探索 LSTM 和 GRU，这两种 RNN 变体能够学习长期依赖关系。</li>

<p dir="auto" _msttexthash="2205065109" _msthash="448">预先训练的模型仅在 next-token 预测任务上进行训练，这就是为什么它们不是有用的助手。SFT 允许您调整它们以响应说明。此外，它允许您在任何数据（私有、GPT-4 看不到等）上微调您的模型并使用它，而无需为像 OpenAI 这样的 API 付费。</p>
<ul dir="auto">
<li _msttexthash="490197240" _msthash="449"><strong _istranslated="1">完全微调</strong>：完全微调是指训练模型中的所有参数。这不是一种有效的技术，但它会产生稍微好一点的结果。</li>
<li _msttexthash="533834795" _msthash="450"><a href="https://arxiv.org/abs/2106.09685" rel="nofollow" _istranslated="1"><strong _istranslated="1">LoRA：</strong></a>一种基于低秩适配器的参数高效技术 （PEFT）。我们只训练这些适配器，而不是训练所有参数。</li>

<div class="markdown-heading" dir="auto"><h3 tabindex="-1" class="heading-element" dir="auto" _msttexthash="25550538" _msthash="460">5. 首选项对齐</h3><a id="user-content-5-preference-alignment" class="anchor" aria-label="永久链接：5. 首选项对齐" href="#5-preference-alignment" _mstaria-label="882999" _msthash="461"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a></div>
<p dir="auto" _msttexthash="1953047954" _msthash="462">在监督微调之后，RLHF 是用于将 LLM 的答案与人类期望保持一致的一个步骤。这个想法是从人类（或人工）反馈中学习偏好，这可用于减少偏见、审查模型或使它们以更有用的方式运行。它比 SFT 更复杂，通常被视为可选。</p>
<ul dir="auto">
<li _msttexthash="436629297" _msthash="463"><strong _istranslated="1">偏好数据集</strong>：这些数据集通常包含多个具有某种排名的答案，这使得它们比指令数据集更难生成。</li>
<li _msttexthash="738668775" _msthash="464"><a href="https://arxiv.org/abs/1707.06347" rel="nofollow" _istranslated="1"><strong _istranslated="1">近端策略优化</strong></a>： 该算法利用奖励模型来预测给定文本是否被人类高度排名。然后，此预测用于优化 SFT 模型，并根据 KL 散度进行惩罚。</li>
<li _msttexthash="1663375181" _msthash="465"><strong _istranslated="1"><a href="https://arxiv.org/abs/2305.18290" rel="nofollow" _istranslated="1">Direct Preference Optimization</a></strong>： DPO 通过将 DPO 重新定义为分类问题来简化流程。它使用参考模型而不是奖励模型（无需训练），并且只需要一个超参数，使其更加稳定和高效。</li>
</ul>

<p dir="auto" _msttexthash="1167943608" _msthash="487">量化是使用较低精度转换模型的权重（和激活）的过程。例如，使用 16 位存储的权重可以转换为 4 位表示。这种技术对于降低与 LLM 相关的计算和内存成本变得越来越重要。</p>
<ul dir="auto">

<li _msttexthash="821418351" _msthash="490"><strong _istranslated="1">GPTQ 和 EXL2</strong>：<a href="https://arxiv.org/abs/2210.17323" rel="nofollow" _istranslated="1">GPTQ</a>，更具体地说，<a href="https://github.com/turboderp/exllamav2" _istranslated="1">EXL2</a> 格式提供了令人难以置信的速度，但只能在 GPU 上运行。模型也需要很长时间才能被量化。</li>
<li _msttexthash="444305550" _msthash="491"><strong _istranslated="1">AWQ</strong>：这种新格式比 GPTQ 更准确（更低的困惑度），但使用的 VRAM 要多得多，而且不一定更快。</li>
</ul>

<div class="markdown-heading" dir="auto"><h3 tabindex="-1" class="heading-element" dir="auto" _msttexthash="6570590" _msthash="525">2. 构建 Vector Storage</h3><a id="user-content-2-building-a-vector-storage" class="anchor" aria-label="永久链接：2. 构建 Vector Storage" href="#2-building-a-vector-storage" _mstaria-label="1040793" _msthash="526"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a></div>
<p dir="auto" _msttexthash="1492839751" _msthash="527">创建矢量存储是构建检索增强生成 （RAG） 管道的第一步。文档被加载、拆分，并使用相关块来生成矢量表示（嵌入），这些向量表示（嵌入）被存储起来以备将来在推理期间使用。</p>
<ul dir="auto">
<li _msttexthash="963177436" _msthash="528"><strong _istranslated="1">摄取文档</strong>：文档加载器是方便的包装器，可以处理多种格式：PDF、JSON、HTML、Markdown 等。他们还可以直接从一些数据库和 API（GitHub、Reddit、Google Drive 等）中检索数据。</li>
<li _msttexthash="907794108" _msthash="529"><strong _istranslated="1">拆分文档</strong>：文本拆分器将文档分解为更小的、语义上有意义的块。与其在 <em _istranslated="1">n</em> 个字符后拆分文本，不如按标题或递归方式拆分，并带有一些额外的元数据。</li>
<li _msttexthash="666696992" _msthash="530"><strong _istranslated="1">嵌入模型</strong>：嵌入模型将文本转换为矢量表示。它允许对语言进行更深入、更细致的理解，这对于执行语义搜索至关重要。</li>
<li _msttexthash="1014462514" _msthash="531"><strong _istranslated="1">向量数据库</strong>：向量数据库（如 <a href="https://www.trychroma.com/" rel="nofollow" _istranslated="1">Chroma</a>、<a href="https://www.pinecone.io/" rel="nofollow" _istranslated="1">Pinecone</a>、<a href="https://milvus.io/" rel="nofollow" _istranslated="1">Milvus</a>、<a href="https://faiss.ai/" rel="nofollow" _istranslated="1">FAISS、</a><a href="https://github.com/spotify/annoy" _istranslated="1">Annoy</a> 等）旨在存储嵌入向量。它们能够根据向量相似性高效检索与查询“最相似”的数据。</li>
</ul>

<div class="markdown-heading" dir="auto"><h3 tabindex="-1" class="heading-element" dir="auto" _msttexthash="21408465" _msthash="537">3. 检索增强一代</h3><a id="user-content-3-retrieval-augmented-generation" class="anchor" aria-label="永久链接：3. 检索增强生成" href="#3-retrieval-augmented-generation" _mstaria-label="1398254" _msthash="538"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a></div>
<p dir="auto" _msttexthash="713599809" _msthash="539">借助 RAG，LLM 可以从数据库中检索上下文文档，以提高其答案的准确性。RAG 是一种无需任何微调即可增强模型知识的流行方法。</p>
<ul dir="auto">
<li _msttexthash="792743341" _msthash="540"><strong _istranslated="1">编排器</strong>：编排器（如 <a href="https://python.langchain.com/docs/get_started/introduction" rel="nofollow" _istranslated="1">LangChain</a>、<a href="https://docs.llamaindex.ai/en/stable/" rel="nofollow" _istranslated="1">LlamaIndex</a>、<a href="https://github.com/IntelLabs/fastRAG" _istranslated="1">FastRAG</a> 等）是流行的框架，用于将 LLM 与工具、数据库、内存等连接起来并增强其功能。</li>
<li _msttexthash="668476692" _msthash="541"><strong _istranslated="1">检索器</strong>：用户说明未针对检索进行优化。可以应用不同的技术（例如，多查询检索器、<a href="https://arxiv.org/abs/2212.10496" rel="nofollow" _istranslated="1">HyDE</a> 等）来改写/扩展它们并提高性能。</li>
<li _msttexthash="1283928295" _msthash="542"><strong _istranslated="1">记忆</strong>：为了记住以前的说明和答案，LLM 和 ChatGPT 等聊天机器人将此历史记录添加到其上下文窗口中。该缓冲液可以通过汇总（例如，使用较小的 LLM）、载体存储 + RAG 等来改进。</li>
<li _msttexthash="713707150" _msthash="543"><strong _istranslated="1">评估</strong>：我们需要评估文档检索（上下文精度和召回率）和生成阶段（忠实度和答案相关性）。它可以通过 <a href="https://github.com/explodinggradients/ragas/tree/main" _istranslated="1">Ragas</a> 和 <a href="https://github.com/confident-ai/deepeval" _istranslated="1">DeepEval</a> 工具进行简化。</li>
</ul>

<p dir="auto" _msttexthash="848842072" _msthash="552">实际应用程序可能需要复杂的管道，包括 SQL 或图形数据库，以及自动选择相关工具和 API。这些高级技术可以改进基线解决方案并提供其他功能。</p>
<ul dir="auto">
<li _msttexthash="1126006947" _msthash="553"><strong _istranslated="1">查询构建</strong>：存储在传统数据库中的结构化数据需要特定的查询语言，如 SQL、Cypher、元数据等。我们可以直接将用户指令翻译成查询，通过查询构造来访问数据。</li>
<li _msttexthash="816423218" _msthash="554"><strong _istranslated="1">代理和工具</strong>：代理通过自动选择最相关的工具来提供答案来增强 LLM。这些工具可以像使用 Google 或 Wikipedia 一样简单，也可以像 Python 解释器或 Jira 一样复杂。</li>
<li _msttexthash="899426905" _msthash="555"><strong _istranslated="1">Post-processing（后处理</strong>）：处理馈送到 LLM 的输入的最后一步。它通过重新排序、<a href="https://github.com/Raudaschl/rag-fusion" _istranslated="1">RAG 融合</a>和分类增强了检索到的文档的相关性和多样性。</li>
<li _msttexthash="257787543" _msthash="556"><strong _istranslated="1">程序 LLM</strong>：<a href="https://github.com/stanfordnlp/dspy" _istranslated="1">DSPy</a> 等框架允许您以编程方式基于自动评估优化提示和权重。</li>
</ul>

<p dir="auto" _msttexthash="724163570" _msthash="566">文本生成是一个成本高昂的过程，需要昂贵的硬件。除了量化之外，还提出了各种技术来最大限度地提高吞吐量并降低推理成本。</p>
<ul dir="auto">
<li _msttexthash="488929961" _msthash="567"><strong _istranslated="1">Flash Attention</strong>：优化注意力机制，将其复杂度从二次转换为线性，从而加快训练和推理速度。</li>
<li _msttexthash="426707853" _msthash="568"><strong _istranslated="1">键值缓存</strong>：了解键值缓存以及<a href="https://arxiv.org/abs/1911.02150" rel="nofollow" _istranslated="1">多查询注意力</a> （MQA） 和<a href="https://arxiv.org/abs/2305.13245" rel="nofollow" _istranslated="1">分组查询注意力</a> （GQA） 中引入的改进。</li>
<li _msttexthash="380505372" _msthash="569"><strong _istranslated="1">推测性解码</strong>：使用小型模型生成草稿，然后由大型模型进行审查，以加快文本生成速度。</li>
</ul>

<p dir="auto" _msttexthash="318908902" _msthash="589">除了与软件相关的传统安全问题外，LLM 由于训练和提示的方式也存在独特的弱点。</p>
<ul dir="auto">
<li _msttexthash="1462385041" _msthash="590"><strong _istranslated="1">Prompt hacking</strong>：与提示工程相关的不同技术，包括提示注入（劫持模型答案的额外指令）、数据/提示泄漏（检索其原始数据/提示）和越狱（制作提示以绕过安全功能）。</li>

</ul>
